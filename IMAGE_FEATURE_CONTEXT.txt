1. PROJECT OVERVIEW
================================================================================

Hermes2 is a native Android Material 3 app that provides a UI for Hermes Agent
(Python AI agent running in Termux) via local WebSocket (ws://127.0.0.1:9119).

- Language: Kotlin 2.0.21
- Framework: Jetpack Compose (Material 3)
- Target Android SDK: 35
- Min Android SDK: 29
- DI: Hilt 2.53.1
- Network: OkHttp (WebSocket)
- JSON: kotlinx.serialization

4. ARCHITECTURE & CONSTRAINTS
================================================================================

STRICT LAYER DEPENDENCY:
  UI Screen → ViewModel → Domain Interface → Implementation (DI-injected)
  - UI must NOT import concrete classes (TermuxBridge, OkHttpGatewayClient)
  - Only ViewModels know about DI

STATE MANAGEMENT:
  - Dialog state MUST be at ChatScreen level (composable scope)
  - Message list in ChatViewModel (StateFlow<List<ChatMessage>>)
  - Streaming text buffered in ChatViewModel.streamingBuffer

IMAGE/FILE FLOW:
  - Images arrive as part of message.delta events (markdown with ![alt](url))
  - URLs are local: file://... or content://... (Termux/app local storage)
  - Files should arrive separately (new RPC event type, needs research)

WEBSOCKET PROTOCOL:
  - JSON-RPC 2.0 over ws://127.0.0.1:9119/api/ws?token=<session_token>
  - Methods: session.create, prompt.submit, session.interrupt, etc.
  - Events: message.delta, tool.start, tool.output, etc.
  - No file/image methods yet defined in Hermes Agent

7. KNOWN ISSUES & DECISIONS
================================================================================

ISSUE: compose-markdown 0.7.x uses Coil 3 internally
  - App uses Coil 2 elsewhere
  - Solution: Don't add custom Coil 2 ImageLoader to MarkdownText
  - Result: Uses library's built-in Coil 3 loader (works fine)
  - Lesson: Trust the library's image loading

ISSUE: LazyColumn + Dialog = BadTokenException
  - Dialog needs composable scope outside the item composition
  - Solution: Lift state (fullscreenImageUrl) to ChatScreen level
  - Result: Dialog can be placed at top-level, recompose safely
  - Already fixed in commit 4685cab

DECISION: Images stay separate from file downloads
  - Images: view in-line or full-screen
  - Files: download to Downloads folder
  - Rationale: Images are transient (agent response), files are artifacts

DECISION: No image upload to chat input yet
  - Future feature: select image from gallery/camera before sending message
  - Current: Only file send (documents, images as files)
  - Reason: Complexity (image processing, threading)

11. ADDENDUM 2026-07-02 — GATEWAY API RESEARCH DONE + PHASE 2 IMPLEMENTED
================================================================================

RESEARCH RESULT (hermes-agent 0.18.0 source, via PyPI sdist):
The gateway ALREADY has everything needed. Nothing to build on the Python side.

  Inbound (app → agent), JSON-RPC over the existing WebSocket:
  - image.attach_bytes {session_id, content_base64, filename}
      → writes to ~/.hermes/images, queues on session, auto-included in the
        NEXT prompt.submit (native vision). Cap 25 MB. Returns {path,...}.
  - file.attach {session_id, data_url: "data:<mime>;base64,<b64>", name}
      → stages file in workspace, returns ref_text "@file:<path>" that must be
        appended to the prompt text.
  - pdf.attach, image.detach also exist.

  Outbound (agent → app), HTTP on the same 127.0.0.1:9119 server:
  - GET /api/files/download?path=<abs>&token=<session_token>
      → streams raw file bytes; token accepted as QUERY PARAM (the only such
        route, see _QUERY_TOKEN_API_PATHS). Works directly as a Coil model URL
        and with DownloadManager. Path policy: home-rooted (Termux), so
        ~/.hermes/images/... is reachable.
  - GET /api/media?path=... → base64 data_url JSON, header-auth only,
        restricted to ~/.hermes/{images,screenshots,cache}. NOT used (download
        endpoint is simpler).

SECURITY DECISION: no termux-setup-storage / shared storage. All file transfer
goes through the loopback gateway only. Termux sandbox stays intact. The only
file crossing the boundary is the one the user explicitly picked (SAF).

IMPLEMENTED (this commit):
  - ChatViewModel.resolveMediaUrl(): local/file:// paths in ![..](..) →
    http://127.0.0.1:9119/api/files/download?path=..&token=..
  - ChatViewModel.attachFromUri(): SAF picker Uri → image.attach_bytes or
    file.attach; pending list in ChatUiState.pendingAttachments.
  - ChatViewModel.removeAttachment(): image.detach for queued images.
  - sendMessage(): appends @file: refs to outgoing prompt; images ride along
    automatically; bubble shows 📎 names.
  - InputBar: real file picker (GetContent "*/*"), attach spinner, staged-
    attachment chips (tap = remove). "Coming soon" toast removed.
  - MessageBubble: inlineImages now mapped through resolveUrl.
